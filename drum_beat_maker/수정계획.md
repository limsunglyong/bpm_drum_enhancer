# Focustone DAW — Drum Beat Maker 시퀀서 정렬 버그 수정 계획 (방안 A 확정)

## 1. 검증으로 확정된 사실

뷰포트 1440·1280px에서 Steps/Bar 3종 × Bars 4종 × Time Signature 4종 = 96조합,
악기 트랙 1~10개를 CSS Grid/Flex 사양 기반으로 시뮬레이션한 결과:

| 버그 | 상태 | 오차 |
|---|---|---|
| BAR 줄 ↔ 패드 (가로) | **정상** | 96/96 조합 0.00px |
| StepMarkers ↔ 패드 (가로) | **버그** | 96/96 조합 238~272px |
| 라벨 ↔ 패드 (세로) | **버그** | 행당 3~32px, 트랙 수만큼 누적 (10개에서 최대 200px) |

Time Signature는 `stepsPerBeat`만 바꾸며 칸 폭·오프셋에 영향이 없다(전 조합 영향 0 확인).

## 2. 근본 원인

### 원인 A — StepMarkers 좌측 스페이서 누락
`components.jsx`의 `StepMarkers`는 좌측 스페이서가 없다(시작 X=0px). BAR 줄과 TrackRow 패드는 둘 다 272px 스페이서를 둔다. 그래서 마커 그리드가 통째로 272px 왼쪽으로 쏠린다.

### 원인 B — 레이어별 minWidth 컨테이너 불일치
BAR 줄과 패드 블록은 `minWidth: 280 + totalSteps*40` 컨테이너 안에 있어 패드가 `minmax(40px,1fr)`에서 40px로 클램프된다. 그러나 StepMarkers는 그 컨테이너 밖이라 부모 폭 전체로 늘어나 칸 폭이 40~96px이 된다. 같은 `repeat(totalSteps,1fr)`인데도 1fr 분배 기준 폭이 달라 칸 폭까지 불일치한다.

### 원인 C — 라벨 이중 렌더링 (구조적 결함, Y축 버그의 원인)
트랙 라벨이 두 곳에서 그려진다.
- 스크롤 레이어: `TrackRow`(패드+라벨 동시) — 행 높이 = `12 + max(라벨, 패드칸폭)`
- absolute 오버레이: `TrackRow`(`labelOnly`, 라벨만) — 행 높이 = `12 + 라벨`

행 높이를 패드(`aspect-square`)가 결정하는데 오버레이는 그 높이를 알 수 없다. 패드칸폭이 라벨 콘텐츠 높이보다 크면 행마다 오차가 생기고 아래로 갈수록 누적된다. 오버레이 방식 자체가 구조적으로 정렬을 보장할 수 없다.

원인 A·B·C는 모두 "라벨/마커를 본체와 분리해 따로 그린다"는 동일한 설계에서 나왔다.

## 3. 수정 방침 — 방안 A 확정

**absolute 라벨 오버레이를 완전히 폐기하고, 라벨을 본체 행 안에서 한 번만 그린다. 가로 스크롤 시 라벨 고정은 `position: sticky; left: 0`으로 처리한다.**

방안 A를 택한 이유:
- 라벨과 패드가 *물리적으로 같은 행*에 있으므로 행 높이가 원천적으로 동일 → Y축 누적 오차가 발생할 수 없다(트랙 수·패드칸폭·줄높이와 무관하게 0px).
- 이중 렌더링이 X축·Y축 버그의 공통 원인이므로, 이를 제거하면 두 축이 동시에 해결된다.
- 선택 하이라이트 중복(3겹)도 라벨이 한 번만 그려지면서 자동 정리된다.
- DAW UI에서 표준적으로 쓰는 방식이다.

(대안 B "행 높이 고정 상수"는 패드 `aspect-square`가 깨져 기각, 대안 C "JS로 높이 측정 동기화"는 이중 렌더링을 그대로 두는 우회라 기각.)

핵심 원칙 3가지:
1. 트랙 헤더 폭을 단일 상수로 정의하고 모든 레이어가 공유한다.
2. BAR 줄·StepMarkers·패드 행을 모두 같은 minWidth 컨테이너 안에 넣어 1fr 분배 기준 폭을 일치시킨다.
3. 라벨은 본체 `TrackRow` 안에서만 그리고, 가로 스크롤 고정은 `sticky`로 한다(오버레이 복제 금지).

## 4. 구체적 변경 내용

### 4.1 공통 상수 도입 (app.jsx)
- 트랙 헤더 폭 상수(예: `LABEL_W = 252`)를 한 곳에 정의.
- 좌측 패딩(`px-2`, `gap-3` 등)으로 생기는 오프셋을 상수에 흡수하거나 모든 레이어에서 동일하게 적용해, 세 레이어의 그리드 시작 X를 단일 값으로 통일.

### 4.2 라벨 오버레이 제거 (app.jsx) — 방안 A 핵심
- `app.jsx`의 absolute 라벨 오버레이 블록(가짜 BAR 스페이서 + 28px div + `labelOnly` TrackRow 복제) 전체를 **삭제**.
- 스크롤 레이어의 `TrackRow` 하나만 남기고, 그것이 라벨과 패드를 같은 행에 함께 그리도록 한다.
- 오버레이 제거로 가로 스크롤 시 라벨이 패드와 함께 밀리게 되므로, **4.4의 `sticky` 처리로 라벨을 고정**한다.

### 4.3 StepMarkers 수정 (components.jsx)
- StepMarkers에 좌측 스페이서(헤더 폭 상수와 동일)를 추가해 BAR 줄·패드와 그리드 시작점을 일치.
- StepMarkers를 BAR 줄·패드와 같은 minWidth 컨테이너 안으로 이동해 1fr 분배 기준 폭을 일치.

### 4.4 TrackRow 정리 (components.jsx) — 방안 A 핵심
- `labelOnly` / `padOnly` 분기 props 제거(오버레이 폐기로 불필요).
- 라벨 div를 본체 행 안에 두되 `position: sticky; left: 0; z-index`를 적용해 가로 스크롤 시에도 화면 왼쪽에 고정.
- 라벨 div에 불투명 `background`를 지정해, 스크롤된 패드가 라벨 뒤로 비치지 않도록 한다.
- 선택 하이라이트를 3겹(루트 `bg-white`, 라벨 그라데이션, 바깥 `ring`)에서 1겹으로 통일. 라벨 배경 + `inset` 보더만 유지.

### 4.5 BAR 줄 (app.jsx)
- 현재 정렬은 정상이므로 폭/오프셋 로직은 유지.
- StepMarkers·패드와 동일 컨테이너·동일 좌측 상수를 쓰도록 정리만 적용.

## 5. 방안 A 리스크 및 대응

- **`sticky` 동작 조건:** `position: sticky`는 스크롤 조상(`overflow-x-auto`)을 기준으로 동작한다. 라벨과 그 사이 조상 어디에도 `overflow:hidden`이 없어야 한다. 현재 `overflow-x-auto` 구조와는 맞물리지만, 수정 시 중간 래퍼의 overflow 속성을 점검한다.
- **stacking/배경:** 라벨이 sticky로 패드 위에 겹칠 때 패드가 비치지 않도록 불투명 배경과 `z-index`를 확실히 준다.
- **재생 플레이헤드 오버레이:** 별도의 playhead absolute 오버레이(app.jsx)는 패드 그리드 기준이므로, 라벨이 sticky로 그 위를 덮을 때 `z-index` 순서를 점검한다.

# 9. 작업 로그 (세션 2) — ring 정렬 버그 정밀 분석 완료, 승인 대기

## 9.1 DOM 중첩 구조 (코드에서 확정)
```
div.overflow-x-auto.pb-1                              ← 가로 스크롤 컨테이너
 ├─ div.flex.mb-1.gap-3.px-2  (style: minWidth O)     ← BAR 줄
 ├─ <StepMarkers/>            (내부 flex.gap-3.px-2)  ← 스텝 마커
 └─ div.mt-2.space-y-0.5      (style: minWidth O)     ← 패드블록 부모 [문제지점]
     └─ div.group  (style: minWidth O, ring 적용)     ← 선택 ring div
         └─ TrackRow 루트 div.flex.gap-3.px-2         ← 라벨+패드 그리드
```

## 9.2 ring 버그 — 진짜 원인 (확정)
- ring은 `group` div 경계에 그려진다.
- `group` div, 그 부모 `div.mt-2.space-y-0.5` 둘 다 **block 요소**다. block 요소의 `width`는 부모 width를 따르며, `minWidth`는 부모 width보다 클 때만 작동한다.
- `div.mt-2.space-y-0.5`는 `width` 없이 `minWidth`만 있다 → 실제 `width`는 `overflow-x-auto`의 **보이는 폭(viewport 폭)**으로 잡힌다.
- 패드 그리드(TrackRow 내부, flex-1 + minmax(40px,1fr))는 콘텐츠가 minWidth(280+totalSteps*40)까지 늘어난다.
- 결과: 패드 그리드 실제 폭 > `group` div 폭 → 패드가 `group`을 오버플로 → ring이 패드 끝까지 못 미치고 **마지막 2칸이 ring 밖으로 빠져나옴.**
- **세션 1에서 `group`에만 minWidth를 준 수정이 효과 없던 이유**: 부모 `div.mt-2`가 여전히 viewport 폭으로 좁게 잡혀, 자식 `group`의 minWidth가 부모 안에서 제대로 확장되지 못함. minWidth는 한 곳만 줘서는 체인 전체가 늘어나지 않는다.

## 9.3 개선 방향 (승인 요청)
핵심: 스크롤 컨테이너 안의 레이어들이 **콘텐츠 폭(max-content)**을 갖게 하여, ring div가 패드 그리드와 같은 폭이 되도록 한다.

구체 변경 (app.jsx):
1. `div.mt-2.space-y-0.5` (패드블록 부모): `width: max-content` 부여. (Tailwind `w-max`)
   → 이 div가 패드 그리드 실제 폭을 갖게 됨.
2. `div.group` (ring div): 부모가 max-content가 되면 `group`도 콘텐츠 폭을 따르도록 `width: max-content` 부여. minWidth 스타일은 유지 또는 max-content로 대체.
3. BAR 줄 div, StepMarkers: 동일하게 `width: max-content` 처리하여 세 레이어가 같은 스크롤 폭을 공유 → 가로 정렬 유지.
4. 위 변경 후에도 세 레이어의 `minWidth: 280+totalSteps*40`는 "창이 콘텐츠보다 넓을 때 최소 폭 보장" 역할로 유지.

검증:
- 수정 후 brower에서 트랙 선택 시 ring이 마지막 패드 셀에 정확히 끝나는지 육안 확인.
- 가로 스크롤·StepMarkers·BAR 줄 정렬이 깨지지 않는지 재확인.
- 정상 확인되면 ring 색상 yellow → cyan 원복, 라벨 배경도 cyan 원복.

## 9.4 상태
- 분석 완료. 코드 미수정. **승인 대기.**

# 10. 작업 로그 (세션 3) — 진짜 원인 확정: minWidth 공식 오류

## 10.1 확정된 근본 원인
모든 레이어의 `minWidth` 공식 `280 + totalSteps*40`이 **틀렸다.**
- TrackRow 실제 콘텐츠 폭 = px-2(16) + 라벨(252) + gap-3(12) + 패드칸(ts*40) + **패드 칸 사이 gap (ts-1)*cellGap**
- 현재 공식엔 마지막 항 `(ts-1)*cellGap`이 누락됨.
- 부족분: ts=8→42px, ts=16→90px, ts=32→124px, ts=64→189px.
- 결과: 패드 그리드(minmax(40px,1fr), 40px 미만 축소 불가)가 컨테이너를 오버플로 →
  ring이 패드 끝에 못 미치고 마지막 칸들이 밖으로 빠짐. + 불필요한 가로 스크롤 발생.

## 10.2 세션 1·2가 실패한 이유 (반복 금지)
- w-max/max-content 시도: 패드 1fr 확장을 죽여 버그2·3 유발 → 원복함.
- ring을 group→TrackRow로 옮긴 시도: 폭 공식이 틀린 한 어디에 ring을 걸어도 오버플로는 동일하게 발생. 위치 문제가 아니라 **폭 공식 문제**였음.

## 10.3 수정 방향 (확정)
- minWidth 공식을 `280 + ts*40 + (ts-1)*cellGap` 로 정정.
- 적용 위치: BAR 줄, 패드블록 부모(div.mt-2), group div. (StepMarkers는 minWidth 미사용 — 점검 후 동일 적용)
- ring 위치는 group div로 되돌려도 무방(폭이 맞으면 group=패드폭).
- 공식 정정 후 w-max 등 임시방편 전부 제거된 상태 유지.

# 11. 작업 로그 (세션 4) — 라벨 Y축 정렬 및 PAD 횡축 스크롤바 두께 수정 반영

## 11.1 문제점 요약

### 라벨 ↔ PAD Y축 정렬 문제
- 기존 구조는 트랙 라벨을 두 번 렌더링했다.
  - 실제 스크롤 레이어: `TrackRow`가 라벨과 PAD를 함께 렌더링.
  - 고정 라벨 레이어: `app.jsx`의 absolute overlay가 `TrackRow labelOnly={true}`로 라벨만 다시 렌더링.
- PAD 행 높이는 `aspect-square` PAD 셀 크기에 영향을 받지만, overlay의 label-only 행은 PAD 셀을 포함하지 않는다.
- 그래서 실제 PAD 행과 overlay 라벨 행이 서로 다른 기준으로 높이를 계산하고, 트랙이 아래로 갈수록 Y축 오차가 누적될 수 있다.
- `rowHeights` + `ResizeObserver`로 실제 행 높이를 측정해 overlay에 맞추려 했지만, 이 방식은 이중 렌더링 구조를 유지하는 우회라 근본 해결이 아니다.

### PAD 하단 횡축 스크롤바 두께
- 전역 스크롤바 스타일은 `drum-beat-maker.html`의 `::-webkit-scrollbar { width: 6px; height: 6px; }`에 정의되어 있었다.
- 사용자가 원하는 변경은 PAD 시퀀서 영역의 횡축 스크롤바 높이를 현재 6px에서 12px로 늘리는 것이다.
- 전역 `height`를 12px로 바꾸면 앱 전체의 가로 스크롤바가 영향을 받을 수 있으므로, PAD 시퀀서 스크롤 컨테이너에만 class를 부여하는 방식이 더 안전하다.

## 11.2 수정 방향

### 라벨 고정 방식
- absolute 라벨 overlay를 제거하고, 실제 `TrackRow` 안의 label만 사용한다.
- label과 PAD가 같은 flex row에 있으므로 행 높이 계산 기준이 하나로 통일된다.
- 가로 스크롤 시 label 고정은 label div에 `position: sticky; left: 0`을 적용해 처리한다.
- label에 불투명 배경과 높은 `z-index`를 유지해, 스크롤된 PAD가 label 뒤로 비치지 않도록 한다.

### 스크롤바 두께
- PAD 시퀀서의 `overflow-x-auto` 컨테이너에 `sequencer-scroll` class를 추가한다.
- `drum-beat-maker.html`에서 `.sequencer-scroll::-webkit-scrollbar { height: 12px; }`만 추가해 PAD 영역 횡축 스크롤바만 2배 두껍게 만든다.
- 기존 전역 스크롤바 `width: 6px; height: 6px;`는 유지한다.

## 11.3 실제 패치 내용

### `app.jsx`
- React destructuring에서 더 이상 사용하지 않는 `useLayoutEffect` 제거.
- `padRowRefs`, `rowHeights`, `ResizeObserver` 기반 행 높이 동기화 로직 제거.
- PAD 시퀀서 스크롤 컨테이너:
  - 기존: `className="overflow-x-auto pb-1"`
  - 변경: `className="sequencer-scroll overflow-x-auto pb-1"`
- 트랙 row map에서 `ref={(el) => padRowRefs.current[index] = el}` 제거.
- absolute 라벨 overlay 블록 전체 삭제.
  - 삭제 대상: `absolute top-0 left-0 z-20` overlay, BAR spacer, StepMarkers spacer, `labelOnly={true}` TrackRow 복제 렌더링.

### `components.jsx`
- `TrackRow` props에서 `labelOnly`, `padOnly` 제거.
- TrackRow 루트 class를 단일 형태로 정리:
  - 기존: `padOnly` 여부에 따라 `pr-2` 또는 `gap-3 px-2`.
  - 변경: 항상 `gap-3 px-2`.
- label div에 sticky 적용:
  - 기존: `track-label ... relative z-10 ...`
  - 변경: `track-label sticky left-0 z-20 ...`
- PAD grid 조건부 렌더링 제거:
  - 기존: `{!labelOnly && <div ...> ... </div>}`
  - 변경: 항상 `<div ...> ... </div>` 렌더링.

### `drum-beat-maker.html`
- PAD 시퀀서 전용 스크롤바 높이 추가:

```css
.sequencer-scroll::-webkit-scrollbar { height: 12px; }
```

## 11.4 기대 효과

- 라벨과 PAD가 같은 `TrackRow` 안에서 한 번만 렌더링되므로 Y축 align 오차가 구조적으로 사라진다.
- 가로 스크롤 시 PAD 셀만 움직이고 label은 왼쪽에 sticky로 고정된다.
- 라벨 overlay, 높이 측정, label-only 분기 등이 사라져 렌더링 구조가 단순해진다.
- PAD 하단 횡축 스크롤바만 12px로 두꺼워지고, 앱 전체 스크롤바 스타일은 기존과 동일하게 유지된다.

## 11.5 검증 상태

- `rg`로 `rowHeights`, `padRowRefs`, `labelOnly`, `padOnly`, `useLayoutEffect`, `absolute top-0 left-0` 잔여 참조가 제거된 것을 확인.
- `git diff --check` 통과.
- 현재 세션에서 in-app browser 연결 대상이 없어 실제 화면 스크린샷 검증은 수행하지 못함.
